# KOSIS 웹앱 기획 프롬프트 (AI용)

아래는 기존 파이썬 배치 프로젝트를 웹앱으로 전환하기 위한 요구사항이다.  
당신은 시니어 솔루션 아키텍트 + 프로덕트 매니저 역할을 수행한다.

## 프로젝트 컨텍스트
- 기준일: 2026-05-11
- 현재 구조:
  - `runner.py`: KOSIS Open API 호출/가공/엑셀 저장 핵심 로직
  - `jobs/`: 카테고리별 JSON 잡 정의
  - `output/`: 결과 xlsx 산출물
  - `run_*.bat`: 배치 실행 진입점
  - `app.py`: Streamlit UI 초안(현재는 스캐폴딩 중심)

## 목표
- 배치 중심 실행을 웹 기반 운영 콘솔로 전환
- 비개발자도 클릭만으로 잡 실행, 상태 확인, 결과 다운로드 가능
- 운영 안정성(재시도, 에러 가시성, 이력)을 강화

## 원하는 결과물
다음 산출물을 순서대로 작성하라.

1. 제품 기획서(PRD)
- 문제 정의, 목표/비목표, 사용자 페르소나, 사용자 시나리오
- 기능 요구사항(Must/Should/Could)
- 화면 구성과 사용자 흐름

2. 기술 아키텍처 제안
- 권장 스택 비교:
  - A안: Streamlit 고도화
  - B안: Next.js + FastAPI
- 권장안 1개 선정 + 선정 근거(복잡도/확장성/개발속도/운영성)
- 시스템 구성도(텍스트 기반)

3. API 설계 초안
- 필수 엔드포인트:
  - 잡 목록 조회
  - 실행 시작(전체/선택)
  - 실행 상태 조회
  - 로그 스트림 조회
  - 결과 파일 목록/다운로드
- 요청/응답 예시(JSON)
- 에러 코드 규칙

4. 데이터/도메인 모델
- `Job`, `Run`, `RunLog`, `Artifact` 중심 스키마 제안
- 상태 전이 정의 (`PENDING -> RUNNING -> SUCCESS|FAILED|CANCELED`)

5. 구현 로드맵
- 3단계(빠른 MVP -> 안정화 -> 확장)
- 각 단계별 기간, 완료 기준(DoD), 리스크와 대응

6. 운영 가이드
- 로그/모니터링 전략
- 장애 대응 시나리오
- 보안 체크리스트(API 키, 파일 접근, 감사로그)

## 출력 형식 규칙
- 한국어로 작성
- Markdown 형식
- 각 섹션은 바로 실행 가능한 수준의 구체성 유지
- 표를 사용할 때는 "결정 기준/트레이드오프"가 드러나야 함
- 마지막에 "추천 실행 순서 TOP 5"를 번호 목록으로 제시

## 추가 제약
- 기존 `runner.py`는 최대한 재사용하고, 점진적 리팩터링을 우선한다.
- 초기 배포는 로컬/사내망 환경을 우선 고려한다.
- 운영자가 개발자가 아닐 수 있음을 가정한다.
